# CallVault — Audit 2026-05-06

Static-only audit. No build was run. Findings derived from reading the recently-touched surfaces called out by Mahendra plus a focused sweep of nav, ViewModels, and security-relevant code. Severity legend: **CRIT** (data loss / user blocked / security) · **HIGH** (broken UX / dead feature) · **MED** (drift / state debt) · **LOW** (cleanup).

> Reconcile this list against `git log --since=2026-05-06 --oneline` before starting work — entries near the top of the list are most likely to already be fixed.

## 1. Bugs, dead code, broken nav

### CRIT-1 — Sign-up and "Forgot password?" are no-op links
**File**: `ui/navigation/CallVaultNavHost.kt:148–158`
The composable for `Destinations.Login` passes empty lambdas:
```kotlin
LoginScreen(
    onAuthenticated = { … },
    onForgotPassword = {},   // ← no-op
    onCreateAccount = {},    // ← no-op
)
```
A complete secondary auth graph exists in `ui/screen/auth/AuthNavGraph.kt` (Welcome, Signup, Forgot, Reset, VerifyEmail) — it's wired to `AuthDestinations.GRAPH` but **never plugged into the main `CallVaultNavHost`**. Result: a brand-new user has no way to create an account from the running app, and no way to recover a forgotten password.

**Fix**: replace the `Destinations.Login` composable block with a `navigation(...)` sub-graph using `authGraph(navController, onAuthenticated = …)`, OR add explicit composable entries for Welcome/Signup/Forgot/Reset/VerifyEmail and wire the two empty lambdas.

### CRIT-2 — `data_extraction_rules.xml` predates the cloud pivot
`DECISIONS.md` ("Sprint 0 — Build verification" section) records that `data_extraction_rules.xml` "excludes every domain (cloud + device transfer) so user data never leaves the device". On 2026-05-05 the `Don'ts` in `CLAUDE.md` were reversed: Supabase Auth, PostHog, FCM are now in scope. The extraction rules file was almost certainly not revisited. **Confirm** whether it still globally blocks cloud backup and decide explicitly: keep blocked (preferred for sideloaded privacy posture) and document, or selectively allow `SettingsDataStore` to participate. Either way, write a paragraph in `DECISIONS.md` "Cloud pivot" so this doesn't drift again.

### HIGH-1 — Export wizard pretends to support 5 formats but only renders Excel
**File**: `ui/screen/export/steps/FormatStep.kt:25–57`
Top of the composable force-resets the selected format to `Excel`:
```kotlin
LaunchedEffect(Unit) {
    if (selected != ExportFormat.Excel) onSelect(ExportFormat.Excel)
}
…
listOf(ExportFormat.Excel).forEach { fmt ->   // ← only Excel
```
yet the `when (fmt)` description block has dead branches for `Csv`, `Pdf`, `Json`, `Vcard`. Worse, `ExportScreen.kt:56` still keys behavior off `state.format == Csv || Excel` for the optional `ColumnsStep`. So:
- The user sees a single Excel "card" with a forced-selected state — looks broken.
- The wizard step indexing logic has a dead branch reachable only with Csv (`showColumns = true`, but no UI ever lets the user pick Csv).

**Fix (one of)**: (a) ship Excel-only intentionally — collapse the step to a single read-only line "Format: Excel workbook" and remove the `LaunchedEffect`/`forEach` charade; (b) re-enable the other formats the exporter classes already support (`data/export/` has the implementations).

### HIGH-2 — Auto-tag rule editor makes the user type raw `tagId` longs
**Files**: `ui/screen/autotagrules/components/ConditionRow.kt:188–199` (`TagApplied`, `TagNotApplied`), `ui/screen/autotagrules/components/ActionRow.kt:74–79` (`ApplyTag`).
The "Has tag" / "Missing tag" / "Apply tag" editors render an `InlineTextField` keyboard-typed `Number` with placeholder "Tag ID". The user is expected to know primary-key longs — unusable. A `TagPickerSheet` already exists (`ui/screen/tags/TagPickerSheet.kt`, 154 lines) with the right UX; it just isn't wired here.

**Fix**: replace those three `InlineTextField` calls with a button that opens `TagPickerSheet` and resolves to a `tagId`. Also display the resolved tag name (chip) so the rule reads as "Has tag · Hot lead" rather than "Has tag · 47".

### HIGH-3 — `MoreScreen` declares `onOpenQuickExport` but never invokes it
**File**: `ui/screen/more/MoreScreen.kt:74`
```kotlin
onOpenQuickExport: () -> Unit = {},
```
No row in the four `MoreRow` lists triggers it. `MainScaffold.kt:393` passes a real callback (`onOpenQuickExport = { quickExportOpen = true }`) — entirely unused. Quick Export is therefore reachable only from the Home tab "Quick actions" chip row. Decide:
- keep it Home-only and **delete** the dead parameter from `MoreScreen` + the lambda binding in `MainScaffold`, OR
- add a Quick Export row to the "Data" group in More.

### HIGH-4 — `Destinations.FilterPresets` routes to a placeholder
**File**: `ui/navigation/CallVaultNavHost.kt:293–296`
```kotlin
composable(Destinations.FilterPresets.route) {
    BackHandler { toHome() }
    CallsPlaceholderScreen()    // ← stub since Sprint 3
}
```
Either implement filter presets (real feature; appears in `FEATURE_BACKLOG.md`) or remove the route and the `Destinations.FilterPresets` entry.

### MED-1 — `UpdateSettingsViewModel` exposes 5 dead surfaces
**File**: `ui/screen/settings/UpdateSettingsViewModel.kt`
Trimmed `UpdateSettingsScreen` only consumes `lastChecked` + `checkNow()`. The VM still publishes `channel`, `autoCheck`, `state`, and exposes `setChannel`, `setAutoCheck`, `clearSkipped`. No caller. Drop them, or add the channel/auto-check toggles back to the screen — current state implies "we forgot to delete".

Also: nothing now schedules `UpdateCheckWorker` from the screen lifecycle. Confirm scheduling is still kicked from app startup or `SettingsViewModel.setUpdateAlerts`. If not, `auto-check` is silently broken.

### MED-2 — `SettingsViewModel.SettingsUiState` carries 7 dead fields
**File**: `ui/screen/settings/SettingsViewModel.kt:20–46`
After the trim, `SettingsScreen` no longer renders the Auto-Save / Bubble / Popup / Lead-Scoring / Auto-Backup / Auto-Backup-retention / Sync-interval rows. The `SettingsUiState` data class still carries all 7 fields and the VM still composes them out of DataStore (`combine` over 19 flows — 7 unused). The matching setters (`setAutoSaveEnabled`, `setBubbleEnabled`, `setPopupEnabled`, `setLeadScoringEnabled`, `setAutoBackupEnabled`, `setAutoBackupRetention`, `setSyncInterval`) are also unreferenced.

**Fix**: trim the data class, the `combine`, and the unused setters. Keep the underlying `SettingsDataStore` keys — the dedicated sub-pages still use them.

### MED-3 — Auth-state side effect can yank user out of mid-flow
**File**: `ui/navigation/CallVaultNavHost.kt:333–345`
Outside the `NavHost { … }`, a top-level `LaunchedEffect(authState)` listens for `SignedOut` and force-navigates to `Login`. If a token expires while the user is filling Onboarding or PermissionRationale, they get redirected mid-task with no message. Acceptable for v1, but: (a) consider snackbar "Your session expired — please sign in again", (b) the popUpTo(0) wipes the back stack, so they re-do onboarding too.

### MED-4 — Three `Destinations` entries are declared but unused as routes
`Destinations.Calls` / `Destinations.Home` / `Destinations.More` are sealed entries but the actual `MainScaffold` tab routing keys off `MainTabRoute.{Home,Calls,Inquiries,More}` strings. The unused entries invite future wiring bugs (someone navigates to `Destinations.Calls.route` thinking it works, gets a `composable` not-found). Delete or wire.

### LOW-1 — `LoginScreen` doesn't use the Neo design system
**File**: `ui/screen/auth/LoginScreen.kt`
Uses `Scaffold`, `Button`, `CircularProgressIndicator`, `TextButton` (Material 3 raw). Inconsistent with the rest of the app and skips `NeoButton`/`NeoTextField`/`NeoLoader`. Ditto for the other 5 unwired auth screens — verify before wiring.

### LOW-2 — `LeadScoreBadge` lives in `components/neo/` but isn't a primitive
It's a domain widget (knows about `LeadScore`). Move to `ui/components/widgets/` or accept the convention. Cosmetic.

### LOW-3 — `Onboarding → Permission` flow runs only after Login
A first-time user must Sign up before seeing the Welcome tour. Common SaaS pattern is the reverse (intro before signup). Defer; flag as a UX call.

### LOW-4 — `TODO.md` P3 already calls out `LazyColumnItemsScopeShim` dead helper in `StatsScreen.kt`
Carried since Sprint 8. Three-line removal — bundle with whichever stats-chart task lands next.

## 4. UI direction recommendation

**Recommendation: HYBRID — keep Neo for hero surfaces, migrate forms / lists to Material 3 expressive.**

**Why**:

The target user is an Indian SMB owner who lives in WhatsApp, TrueCaller, and Google Phone — all three are Material 3 (Google Phone is the M3 reference app on this content type). A fully neumorphic app reads as "design exercise" to that audience; the soft pillow shadows look unfamiliar on long input forms (Rule editor, Settings, Auth). Conversely, the Home/Calls/Inquiries/More tabs *do* benefit from Neo: dashboard tiles, call rows, lead-score badges and chips genuinely feel premium and differentiated.

**Concrete split**:

| Surface | Stay Neo | Migrate to M3 |
|---------|----------|---------------|
| Home Today card, Stat tiles, Quick actions chips | ✅ | |
| Call rows, UnsavedPinnedSection, BulkActionBar | ✅ | |
| LeadScoreBadge, TagsSection chips | ✅ | |
| HeroCard, ActionBar (in Call detail) | ✅ | |
| Bottom tab bar (NeoTabBar) | ✅ | |
| **Auth screens (Login, Signup, Forgot, Reset)** | | ✅ M3 — these need to feel like every other Android sign-in form |
| **Forms: TextField inputs in RuleEditor, Settings, Auto-save settings** | | ✅ M3 OutlinedTextField — lower visual cost, better keyboard affordance |
| **Dialogs (NoteEditorDialog, TagEditorDialog, BookmarkReasonDialog, NeoDialog)** | | ✅ M3 AlertDialog with M3 Expressive shape morphing — Neo dialogs are a known stress point in the codebase |
| **Settings rows** | | ✅ M3 ListItem — denser, clearer toggle alignment |
| **ExportScreen wizard steps** | | ✅ M3 — wizards live or die on familiarity |

**Pros**:
- Familiarity bump on the surfaces that *should* feel boring (forms).
- Keeps the brand-defining surfaces (Home, Calls, Lead score) Neo — the moments that show up in screenshots and convince a user this is a "good" app.
- Material 3 expressive is the path Google is investing in; staying ahead of that curve is cheap insurance.
- Lower maintenance: 25 Neo components is a lot for a single-engineer project. Cutting forms/dialogs out drops effective surface to ~12.

**Cons**:
- Visual inconsistency risk if not policed. Mitigation: write 1 page in `UI_GUIDE.md` ("Neo vs M3 — when to use which") and treat it as a code-review checklist.
- Migration effort: ~15 screens to revisit. Estimate **2–3 weeks** for a focused pass; can be done sprint-by-sprint as those screens are touched anyway.

**Not recommended**:
- *Stay full Neo + tighten consistency*: still leaves auth/forms looking off-brand for the target audience.
- *Migrate everything to M3*: throws away the only thing that visually distinguishes CallVault from generic dialer apps. The Home tab stat tiles in particular are the screenshot-worthy moment.
